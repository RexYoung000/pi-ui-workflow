# pi-ui-workflow 迭代计划

> 记录时间：2026-06-06  
> 目的：把当前讨论形成的产品化路线固定到项目内，方便后续换 session 后继续推进。

## 当前产品判断

`pi-ui-workflow` 目前是一个本地可用的 UI 工作流模板集合，核心由：

- `.pi/prompts/`：工作流入口命令
- `.pi/agents/`：7 个专业子 Agent
- `README.md` / `docs/usage.md`：使用说明

下一阶段目标不是简单增加功能，而是把它升级为一个更稳定、更可分发、更有产品体验的 Pi UI 工作流扩展。

---

## 总体路线

### V1：MVP 产品闭环

目标：让工作流逻辑稳定、输出质量更可靠、用户等待过程更清楚。

V1 做完后，用户应能清楚知道：

1. 应该选择哪种执行模式；
2. 大概要等多久；
3. 当前执行到哪一步；
4. 每个子 Agent 产出了什么；
5. 最终设计规范如何整合而来。

#### V1 任务

1. **设计执行模式**
   - 在工作流开始阶段让用户选择模式。
   - 模式建议：快速模式 / 标准模式 / 深度模式。
   - 每种模式需要告知：预计耗时、token 成本、输出质量、适用场景、Agent 编排方式。

2. **统一工作流顺序**
   - 统一 `README.md`、`docs/usage.md`、`.pi/prompts/ui-workflow.md`、`.pi/prompts/ui-review.md` 中的流程描述。
   - 避免不同入口执行顺序不一致。
   - 依赖：先确定执行模式。

3. **修正报告传递**
   - 解决 visual / interaction / content 等 Agent 所需上游报告在当前并行执行中拿不到的问题。
   - 与执行模式联动：
     - 快速模式：允许更多并行，换取速度；
     - 标准模式：建议两段式，平衡速度与质量；
     - 深度模式：更严格串行 / 半串行，追求最高一致性。
   - 依赖：先确定执行模式。

4. **增加报告归档**
   - 建议保存到：

```text
docs/ui-workflow/
├── 00-need-summary.md
├── 01-research-report.md
├── 02-need-report.md
├── 03-form-report.md
├── 04-visual-report.md
├── 05-ia-report.md
├── 06-interaction-report.md
├── 07-content-report.md
└── 99-ui-design-spec.md
```

   - 目的：方便追溯、复跑、调试和减少黑盒感。

5. **优化等待反馈**
   - MVP 先做文字型进度反馈，不依赖真正动画。
   - 例如：

```text
✅ 需求摘要已完成
🔍 正在搜索参考素材...
🧠 研究员正在分析素材...
🎨 视觉分析师正在生成 Design Tokens...
⏳ 还剩：交互分析 / 内容策略 / 总审核
```

6. **补充快速上手**
   - 在 README 前部增加“30 秒开始使用”或“最短路径”。
   - 让用户知道如何安装、如何启动、输入什么命令、成功后会看到什么。

### V2：可分发版本

目标：让别人更容易安装、更安全地使用。

#### V2 任务

1. **调整安全默认值**
   - 检查并优化：

```json
{
  "agentScope": "both",
  "confirmProjectAgents": false
}
```

   - 避免分发给别人使用时默认跳过项目本地 Agent 安全确认。

2. **优化安装体验**
   - 增加安装检查脚本或一键安装脚本。
   - 检查内容包括：
     - subagent 是否存在；
     - design-dna 是否存在；
     - prompts 是否安装；
     - agents 是否安装；
     - 是否需要 `/reload`。

3. **探索包化方案**
   - 研究是否可以做成标准 Pi package。
   - 重点确认：
     - prompts 是否可通过 package manifest 分发；
     - agents 是否可直接分发；
     - 如果 agents 不能直接分发，是否需要转成 skill 或 extension 内部资源；
     - 是否暂时保留脚本复制方案。

4. **增加版本记录**
   - 增加 `CHANGELOG.md`。
   - 规范版本号与发布记录。

---

### V3：高级体验版本

目标：提升复用能力、自动化能力和产品质感。

#### V3 任务

1. **设计局部复跑**
   - 支持只重跑视觉、只重跑交互、只重新搜索素材、从已有规范直接生成 HTML 原型等。
   - 依赖：先完成报告归档。

2. **探索动态进度**
   - 研究 Pi prompt template 与 extension 的能力边界。
   - prompt template 可先做阶段性文字进度。
   - 真正的等待动画、状态栏进度条、TUI 面板或动态 Agent 队列展示，可能需要 Pi extension。

## 产品原则

1. **先做能稳定跑通的 MVP 闭环，再做动态动画和包化。**
2. **把等待时间显性化，让用户知道为什么等、还要等多久。**
3. **把质量和速度的取舍交给用户，而不是在 prompt 里写死。**
4. **所有子 Agent 的中间产物都应尽量可追溯。**
5. **默认安全策略要适合分发，而不仅仅适合本地自用。**
